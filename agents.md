## 🛠 Окружение и управление пакетами (Poetry)
- **Менеджер:** Используй только **Poetry**. Никаких `pip` или `requirements.txt`.
- **Установка:** `poetry install`
- **Добавление зависимостей:** 
  1. Сначала предложи библиотеку в плане (Review Step).
  2. После одобрения используй `poetry add <package>`.
- **Запуск команд:** Все команды (pytest, ruff) выполняй через `poetry run`.

## 🔄 Рабочий процесс (Workflow) с ревью
1. **План (Review):** Сформулируй план изменений. Если нужны новые пакеты, укажи их версии для `pyproject.toml`.
2. **Утверждение:** Дождись команды "Go" или "Confirm" от пользователя.
3. **Реализация:** 
   - Обнови зависимости: `poetry add ...` (если нужно).
   - Напиши логику FastAPI (schemas -> core -> api).
4. **Валидация:**
   - `poetry run ruff check .` — проверка стиля.
   - `poetry run mypy .` — проверка типов.
   - `poetry run pytest` — проверка функционала.

## 🏗 Архитектурные требования FastAPI
- **Core-first:** Бизнес-логику размещай в `core/`, а не в роутерах.
- **Interfaces:** Контракты (ports) размещай в `core/interfaces/` (`Protocol` или `ABC`).
- **Repositories:** Реализации интерфейсов размещай в `repositories/` (или `infra/repositories/`).
- **Dependencies:** Внедряй зависимости через `Annotated[InterfaceType, Depends(get_dependency)]` с подстановкой реализации в DI-слое API.
- **Clean Code:** Роуты должны оставаться тонкими: в `api/` только валидация/маппинг/вызов use-case, тяжелая логика — в `core/`.

## 🔎 Ограничение размера изменений
- **Небольшие PR:** По умолчанию делай изменения минимальными и удобными для ревью (без лишних рефакторингов).
- **Целевой размер MR:** До 300 changed lines (additions + deletions, без учета lock/auto-generated файлов) и до 8 файлов.
- **Если лимит превышен:** До начала реализации заранее предупреди, запроси подтверждение и предложи план разбиения на несколько MR.
- **Один MR — одна цель:** Не смешивай feature/fix/refactor в одном MR без явной необходимости.

## 📝 Коммиты
- **Предлагай commit message:** После изменений предлагай имя коммита, которое явно и кратко отражает суть выполненных изменений.
